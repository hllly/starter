# 轻入口系统设计总方案（种子用户版）

## 1. 文档定位

本文档定义一套面向种子用户阶段的轻入口系统设计方案，目标是服务于当前外贸 SOHO 客户发现系统的早期试点使用。  
该方案强调：

- 用户入口足够轻
- 结果回显清楚
- 反馈结构化
- 后端工作流仍由内部控制
- 能逐步沉淀为外贸 SOHO 的业务闭环
- 页面整体看起来像“完整系统当前开放的核心能力”，而不是一个浅层玩具工具

本文档覆盖：

- 产品定位
- 页面信息架构
- 页面结构与字段
- 三层反馈设计
- 页面视觉与配色系统
- 前后端技术架构
- 数据库表设计
- API 接口设计
- 本地 worker 与云端联动方式
- 版本边界与演进建议

---

# 2. 产品定位

## 2.1 当前阶段的产品定义

该产品不是完整 SaaS，也不是公开全量开放平台，而是一个：

**面向种子用户的半托管客户发现与维护工作台**

当前优先开放的核心能力：

1. 目标客户发现
2. 结果分层与反馈
3. 客户维护更新
4. 任务进展与历史查看

暂不全量开放但会逐步演进的能力：

- 自动触达建议
- 高价值客户持续跟进提醒
- 更深层采购信号增强识别
- 询盘/回复闭环
- 更复杂的自动化调度与推荐

---

## 2.2 当前阶段的产品目标

当前阶段不追求：

- 大规模自助化
- 高并发
- 全自动无人值守
- 面向普通大众的大而全产品形态

当前阶段重点追求：

- 少量真实种子用户可持续使用
- 用户输入简洁
- 结果回显清楚
- 反馈可结构化沉淀
- 后端工作流持续迭代
- 平台层与公司层逐步形成数据闭环

---

## 2.3 当前阶段的用户感知目标

用户进入系统后应感受到：

- 这不是一个“随手搜公司”的小工具
- 这是一个完整系统当前优先开放的核心能力
- 当前开放的是最稳定、最有业务价值的一段
- 自己正在参与一个持续迭代中的专业系统

因此，设计上必须避免：

- 过于轻飘的 AI 玩具感
- 只有表单和列表的薄页面
- 大量空白功能造成“没做完”的感觉
- 技术术语过多、让用户暴露在内部复杂性中

---

# 3. 产品信息架构

建议一级导航只保留 4 个页面：

1. **发现**
2. **线索**
3. **维护**
4. **进展**

这四个页面分别对应：

- 发现：任务输入
- 线索：结果回显与线索级反馈
- 维护：客户状态更新
- 进展：历史任务、当前开放能力、路线感

这种结构既轻，又能体现完整流程感。

# 4. 页面级设计

## 4.1 页面一：发现页

### 页面目标

让用户快速提交本轮客户发现任务，并感受到：

- 当前开放的是系统核心能力
- 系统围绕业务目标运作
- 提交后会进入一个完整处理链路

### 页面结构

#### A. 顶部标题区

**标题：**  
客户发现工作台

**副标题：**  
围绕你的目标市场与客户画像，生成本轮重点客户发现任务。  
当前优先开放：客户发现、结果反馈与客户维护。

#### B. 当前开放能力卡片区

三张横向卡片：

##### 卡片 1：目标客户发现
说明：围绕品类、地区和客户类型，筛选更适合当前开发目标的客户对象。

##### 卡片 2：结果分层与反馈
说明：对发现结果进行推荐、观察与维护分类，并支持你快速校正系统判断。

##### 卡片 3：客户维护更新
说明：记录已联系、已回复和持续跟进对象，为后续筛选与推荐提供依据。

#### C. 正在逐步开放区

三张轻卡片，标注状态：

##### 卡片 A：自动触达建议
标签：定向测试  
说明：基于已发现客户，逐步开放更贴近实际开发动作的触达建议。

##### 卡片 B：重点客户持续跟进
标签：逐步开放  
说明：针对重点客户提供更连续的维护和状态追踪支持。

##### 卡片 C：采购信号增强识别
标签：内测中  
说明：进一步增强对高价值合作信号与采购倾向的识别能力。

#### D. 本轮任务表单区

**区块标题：** 创建本轮发现任务

### 字段设计

#### 1. 目标品类
- 类型：文本 / 下拉可输入
- 必填
- 占位：例如：宠物用品、家居用品、文具、电子配件

#### 2. 目标地区 / 市场
- 类型：多选 / 文本
- 必填
- 占位：例如：美国、欧洲、东南亚、中东

#### 3. 理想客户类型
- 类型：多选
- 必填
- 选项建议：
  - 进口商
  - 分销商
  - 批发商
  - 品牌方 sourcing
  - 连锁零售买手
  - 贸易公司

#### 4. 公司规模偏好
- 类型：单选
- 必填
- 选项：
  - 小型
  - 中型
  - 中大型
  - 不限

#### 5. 本轮更想优先找哪类客户
- 类型：单选
- 选填但建议填写
- 选项：
  - 更容易成交的
  - 更强采购信号的
  - 更适合 OEM / ODM 的
  - 更适合长期开发的
  - 更适合分销合作的

#### 6. 排除项
- 类型：多选 + 自定义补充
- 选项建议：
  - 排除纯零售
  - 排除中国同行
  - 排除平台页
  - 排除超大公司
  - 排除本地自产闭环制造商
  - 其他

#### 7. 当前供货/合作能力说明
- 类型：短文本
- 选填
- 示例：
  - 支持 OEM / ODM
  - 可接受 MOQ 范围
  - 主要价格带
  - 交期能力说明

#### 8. 当前主要痛点
- 类型：单选
- 选填
- 选项：
  - 找不到新客户
  - 线索质量差
  - 很多客户不回复
  - 找到的客户不适合我
  - 需要更匹配的买家

#### 9. 补充说明
- 类型：多行文本
- 选填
- 字数建议限制在 300 字以内

#### E. 提交后提示区

标题：提交后会发生什么

文案：

1. 系统会围绕你的目标进行客户发现  
2. 结果会按推荐、观察和维护状态整理  
3. 你的反馈将直接影响后续推荐方向

#### F. 按钮区

- 主按钮：开始本轮发现
- 次按钮：保存为草稿

## 4.2 页面二：线索页

### 页面目标

让用户感受到：

- 系统已经做了判断与分层
- 结果不是简单名单
- 可以快速对线索做判断协同
- 可以给本轮方向反馈

### 页面结构

#### A. 顶部任务摘要条

显示：

- 任务名称
- 品类
- 市场
- 客户类型
- 更新时间

右侧三张数字卡：

- 推荐客户数量
- 观察客户数量
- 维护中客户数量

#### B. 本轮建议摘要区

标题：本轮建议摘要

文案模板示例：

本轮结果更偏向分销商与品牌 sourcing 类对象。  
建议优先处理推荐区中的高匹配客户，观察区可结合你的产品适配度进一步判断。  
当前结果主要来自行业目录与协会类来源。

#### C. 分层结果区

建议用 3 个 Tab 或 3 个区块：

1. 推荐客户
2. 观察客户
3. 维护中

### 每条线索卡片字段设计

#### 基础信息
- 公司名
- 国家/地区
- 网站
- 客户类型标签

#### 系统判断
- 当前层级：推荐 / 观察 / 维护中
- 推荐理由（1~2 句）
- 来源平台 / 来源类型
- 推荐动作

#### 可折叠高级信息（第一版可选）
- 判断依据摘要
- 适配标签
- 备注

### 推荐客户卡片区文案
说明：这些对象与当前任务目标更匹配，建议优先查看与联系。

### 观察客户卡片区文案
说明：这些对象与当前目标部分匹配，建议结合你的产品与合作方式进一步判断。

### 维护中卡片区文案
说明：这些对象已进入联系、回复或持续跟进阶段，可在维护页更新更细状态。

# 5. 三层反馈设计

反馈不定义为“给产品打分”，而定义为三种业务协同：

1. 线索判断协同
2. 本轮方向校正
3. 业务进展更新

## 5.1 第一层：线索级反馈

### 目标
帮助系统校正单条线索的价值判断。

### 放置位置
线索页，每条线索卡片内。

### 字段设计

#### 价值反馈
- 高价值
- 一般
- 无价值

#### 动作反馈
- 已联系
- 暂不联系
- 加入维护

#### 高价值原因（可多选）
- 客户画像匹配
- 采购信号较强
- 更适合我的产品
- 更容易切入
- 其他

#### 无价值原因（可多选）
- 不是目标客户
- 更像同行
- 联系方式/信息太弱
- 更像零售而不是买家
- 公司规模不匹配
- 其他

## 5.2 第二层：本轮反馈

### 目标
帮助系统调整下一轮推荐方向。

### 放置位置
线索页底部，任务级反馈表单。

### 字段设计

#### 1. 这批结果整体更接近哪种状态？
- 基本符合预期
- 有少量高价值
- 方向还不够准
- 需要明显调整

#### 2. 这批结果最大的问题是什么？
多选：
- 客户类型不够准
- 地区偏差
- 公司规模不合适
- 采购信号不强
- 联系方式不足
- 结果偏少
- 重复感偏强
- 其他

#### 3. 下一轮你更想偏向哪类客户？
- 更高匹配度
- 更强采购信号
- 更适合 OEM / ODM
- 更适合长期开发
- 更细子类目
- 更换市场方向

#### 4. 文本补充（选填）
- 让用户补充一两句方向说明

## 5.3 第三层：业务结果反馈

### 目标
形成真实外贸开发闭环。

### 放置位置
维护页为主，也可以在线索卡片中跳转进入。

### 字段设计
- 已发送开发信
- 有回复
- 有效回复
- 已报价
- 有样品机会
- 持续跟进
- 暂停跟进
- 无意向

#### 备注字段
用于记录：
- 回复方向
- 跟进计划
- 暂停原因
- 其他业务说明

# 6. 视觉与配色系统

## 6.1 风格目标

整体风格应体现：

- 清新简约
- 轻专业
- 有系统感
- 不花哨
- 不像 AI 玩具
- 不要性冷淡风
- 高亮色不超过 3 种

## 6.2 配色策略

采用：

**浅青绿 + 暖灰白**

### 配色结构
- 大面积：暖灰白 / 米灰白
- 主色：低饱和浅青绿 / teal
- 辅助状态色：柔和绿色、琥珀、柔和红
- 中性色：用于文字、边框、卡片层级

## 6.3 推荐颜色层级（可作为设计参考）

### 背景色
- 页面背景：暖灰白
- 大模块背景：更浅的暖白
- 内容卡片：白色

### 主色
- 主按钮
- 选中态
- 主要链接
- 推荐标签
- 关键 CTA

### 中性色
- 标题文字：深灰
- 正文文字：中深灰
- 次级文字：灰色
- 边框：浅灰

### 状态色
- 成功 / 维护中：柔和自然绿
- 观察 / 提醒：低饱和琥珀
- 错误 / 无意向：低饱和柔红

## 6.4 配色使用原则

1. 页面 70% 以上区域使用中性色
2. 主色只用于关键交互和强调
3. 状态色尽量克制，不要到处都是彩色标签
4. 不使用高饱和科技蓝或高亮紫
5. 不做大面积纯色背景
6. 不做强渐变
7. 通过留白、字号、边框建立层次，不靠颜色堆砌

# 7. 技术架构方案

## 7.1 总体架构

当前阶段推荐采用：

- **用户前后端上云**
- **openclaw 工作流先不在云上**
- **本地 Python worker 拉取云端任务并回传结果**

即：

### 云端产品层
负责：
- 用户登录/邀请
- 任务提交
- 状态展示
- 结果回显
- 反馈收集
- 任务与结果数据库存储

### 本地执行层
负责：
- 拉取待执行任务
- 调用现有 Python 工作流
- 生成 run 目录与原始结果
- 回传结构化结果摘要

## 7.2 推荐技术栈

### 前端
- Next.js
- Tailwind CSS
- shadcn/ui
- React Hook Form
- Zod

### 后端
- Next.js Route Handlers / Server Actions
- Prisma

### 数据库
- PostgreSQL

### 工作流执行
- Python worker
- 轮询 jobs 表
- 调现有 `workflow_controller.py`

### 原始产物存储
- 继续保留在本地文件系统 run 目录中
- 云端数据库只存结构化摘要和索引

# 10. 本地 worker 与云端联动设计

## 10.1 基本模式

推荐使用：

**本地 Python worker 主动轮询云端任务**

## 10.2 执行流程

1. 用户在云端提交发现任务  
2. 后端创建 `discovery_request` 与 `job`  
3. 本地 worker 轮询 jobs 表或云端接口  
4. 本地 worker 调现有 Python workflow  
5. 本地 worker 将结构化结果回传云端  
6. 用户在页面看到结果并反馈

## 10.3 本地 worker 设计建议

### 第一版原则
- 单 worker
- 串行执行
- 一次只处理一个任务
- 先稳定，再并发

### worker 需要具备的能力
- 轮询待执行任务
- 加锁 / 占用任务
- 调 workflow_controller
- 读取结构化摘要
- 回传结果
- 回写错误状态
- 定期 heartbeat

## 10.4 结果回传原则

不要把整个 run 目录同步到云端。  
云端只保存：

- request / job / run 基础信息
- 本轮结果摘要
- 可展示 lead 列表
- 关键统计
- 原始文件路径索引（可选）

本地保留：

- prompts
- temp files
- logs
- validation reports
- tsv/json 原始产物

# 11. 页面与反馈的数据闭环关系

轻入口系统需要支持这样一条闭环：

用户提交任务  
↓  
后端运行发现系统  
↓  
系统回显结果（推荐 / 观察 / 维护）  
↓  
用户做线索级反馈  
↓  
用户做本轮反馈  
↓  
用户后续更新维护状态  
↓  
这些反馈反哺平台层 / 公司层 / 评分层

# 12. 产品边界与版本策略

## 12.1 第一版明确不做的内容

- 复杂多用户协作
- 用户自己操作 workflow 参数
- 全量自动化调度
- 大规模并发任务系统
- 高复杂度筛选器
- 用户直接接触平台层/公司层内部结构
- 工作流原始文件云端全量可见

## 12.2 第一版优先上线顺序

### 第一批上线
1. 发现页
2. 线索页（含第一层、第二层反馈）

### 第二批上线
3. 维护页（第三层反馈）

### 第三批上线
4. 进展页

## 12.3 用户感知策略

页面必须始终传达：

- 当前开放的是最核心的能力
- 其他能力处于逐步开放中
- 这不是一个薄工具，而是一个持续扩展中的专业系统
- 你的反馈会影响后续开放顺序与推荐逻辑

# 13. MVP 成功标准

在种子用户阶段，这套轻入口系统如果满足以下条件，就算成功：

1. 用户能在 3 分钟内完成任务提交
2. 用户能在结果页理解为什么推荐这些线索
3. 用户愿意对线索做轻反馈
4. 用户能更新至少一部分维护状态
5. 你能从数据库中拿到结构化的任务、线索、反馈、维护数据
6. 用户感受到这是一个工作台，而不是单点找客户工具

# 15. 一句话总结

这套轻入口不是为了替代你的工作流引擎，而是为了给种子用户提供一个：

**“可提交任务、可查看结果、可反馈校正、可更新维护状态”的专业工作台入口。**

其设计原则是：

- 前端轻
- 后端清晰
- 工作流仍由你掌控
- 反馈结构化
- 当前只开放核心能力，但整体呈现上具有完整系统感
